## Карта блока в одном предложении

Розница — это мобильный банк для физлиц: своя БД клиентов и операций, FastAPI-сервис с публичными ручками, маленький мобильный UI, и встроенные точки расширения для кредита и инвестиций (пока пустые).

## Что лежит в папке /retail/

```
retail/
├── pyproject.toml      описание зависимостей сервиса (FastAPI, Postgres-драйвер, httpx)
├── Dockerfile          как собирается контейнер для деплоя
├── src/
│   ├── main.py         все публичные ручки (HTTP API) сервиса
│   ├── db.py           работа с Postgres (схема, миграции, CRUD)
│   └── static/         мобильный UI (одна HTML-страница, рисующая «приложение»)
├── CLAUDE.md           инструкции для AI-агента блока
├── state.md            «живой» документ зоны ответственности
└── ARCHITECTURE.md     этот файл
```

## Публичные ручки (то, что Розница отдаёт наружу)

| Ручка                          | Что делает                                                         | Статус             |
|--------------------------------|--------------------------------------------------------------------|--------------------|
| `GET /health`                  | проверка живости + сколько клиентов и транзакций загружено         | работает           |
| `GET /clients`                 | список клиентов с фильтрами (segment, has_overdue, min_income)     | работает           |
| `GET /clients/{id}`            | карточка одного клиента                                            | работает           |
| `GET /transactions/{id}`       | лента операций клиента                                             | работает           |
| `POST /api/transfer`           | атомарный перевод (внутри банка или «наружу»)                      | работает           |
| `POST /api/credit-apply`       | приём заявки на кредит                                             | заглушка (501)     |
| `GET /credit-applications`     | список заявок на кредит                                            | работает (читает заявки) |
| `POST /_seed/load`             | служебная — догрузить seed                                         | работает           |
| `GET /`, `/static/...`         | мобильный UI «карман»                                              | работает           |

## Мобильный UI (что видит клиент в приложении)

В `src/static/index.html` лежит одностраничное «мобильное приложение» Райф с тремя вкладками: профиль клиента (имя, сегмент, остаток, продукты), переводы (форма с указанием получателя и суммы), операции (история транзакций).

Чего в UI **нет** и что предполагается строить:
- вкладки «Кредит» — это первый разогревочный кейс воркшопа;
- вкладки «Инвестиции» для премиум-клиентов — это часть второго кейса.

## Связи с соседями (по правлению)

Архитектурно блок готов ходить к соседям через переменные окружения `NEIGHBOR_*`. На проде Render подставляет туда настоящие URL, локально — fallback на `http://localhost:<порт_соседа>`. Сейчас фактических вызовов нет, но как только начнём строить кредит и инвестиции — потребуются вот такие маршруты:

```
                                 ┌─────────────────┐
                                 │     CEO         │  собирает дашборд правления
                                 │ raif-ceo:8000   │  /dashboard агрегирует из всех блоков
                                 └────────▲────────┘
                                          │
                                          │ (все блоки → CEO агрегирует наружу)
                                          │
   ┌────────────┐                ┌────────┴────────┐                ┌────────────┐
   │   Risk     │ ◄───скоринг───┤     RETAIL      ├───лимиты──────►│  Finance   │
   │ raif-risk  │                │  raif-retail    │                │raif-finance│
   │   :8050    │  /score/{id}   │     :8020       │  /balance/{id} │   :8040    │
   │ /credit-   │                │                 │  /limits       │            │
   │  history/  │                │                 │  /internal-    │            │
   │  {id}      │                │                 │   transfer     │            │
   └────────────┘                │                 │                └────────────┘
                                 │                 │
   ┌────────────┐                │                 │                ┌────────────┐
   │     IT     │ ◄────LLM──────┤                 ├──инвест-кат.──►│    CIB     │
   │  raif-it   │                │                 │                │  raif-cib  │
   │   :8030    │  /llm/ask      │                 │  /products     │   :8010    │
   │ /llm/audit │                │                 │  /invest/      │            │
   │ /llm/status│                └─────────────────┘   recommend    └────────────┘
   └────────────┘                                                   /corp/payment
```

Что мы можем взять у соседей прямо сейчас:
- **Risk** — `/score/{client_id}` (риск-скор клиента) и `/credit-history/{client_id}` (история просрочек). Понадобится для кредитного решения.
- **Finance** — `/balance/{client_id}` (текущий баланс), `/limits` (лимиты), `/internal-transfer` (внутренний платёж). Понадобится для подтверждения переводов и проверки лимитов.
- **IT** — `/llm/ask` (поход в LLM с указанием `requester="retail"` для аудита). Это единственный способ обращаться к языковой модели — собственного ключа у Розницы нет. Понадобится, чтобы объяснять клиенту понятным языком решение по кредиту.
- **CIB** — `/products` (каталог), `/invest/recommend?client_id=...` (рекомендация инвестинструментов). Понадобится для второго кейса про премиум-инвестиции.
- **CEO** — сам ничего не отдаёт нам, только агрегирует наружу.

## Каналы взаимодействия с другими блоками

1. **Прямой HTTP** — для «горячих» запросов в реальном времени (скоринг клиента в момент заявки, проверка баланса).
2. **INBOX** (`/INBOX/to_<блок>.md`) — для договорённостей: «коллега, нам нужна такая-то ручка», «можешь увеличить лимит до …», «верни вот такой формат». Этот канал **видим всем** и проектируется на табло во время воркшопа.
3. **`/contracts/`** — пустая общая папка для совместных контрактов, если решим зафиксировать общие форматы данных.

## Ограничения (что мы делать не будем)

- Не лезем в код соседей напрямую — у нас на это нет прав, и мы не хотим ломать им работу. Только через INBOX или их публичные ручки.
- Не ходим в OpenAI/внешний LLM напрямую — только через IT.
- Не делаем фейк-моки для соседских ручек, если они «ещё не готовы». Лучше пишем им в INBOX и даём клиенту честный ответ «эта функция временно недоступна».
