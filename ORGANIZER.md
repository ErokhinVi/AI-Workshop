## Архитектура воркшопа

**Один общий monorepo, шесть изолированных папок.** Каждый агент топа физически заперт в своей папке через `.claude/settings.local.json` (см. `.claude/templates/settings-<блок>.json`). Поправить чужой код агент не может — Claude Code откажет.

**Коммуникация между блоками — только через `/INBOX/`.** В нём шесть файлов `to_<блок>.md`. Если CIB-агенту нужно что-то от Risk — он пишет в `to_risk.md`. Эти переписки видны Виталию и Нерсесу на проекторе и обсуждаются в зале — это и есть главный обучающий сюжет дня.

**Общая зона — `/contracts/`.** Сюда складываются договорённости (общие типы данных, OpenAPI-спеки), если участники до них дойдут. Заранее ничего не предзаполнено — структуру выбирают сами.

**Деплой — Render через GitHub Actions.** Каждый блок собирается в отдельный сервис на Render. После `git push` Action `.github/workflows/deploy-render.yml` дёргает Deploy Hook соответствующего сервиса. Соседи общаются через `httpx.get(NEIGHBOR_<BLOCK>/...)` — env-переменные `NEIGHBOR_*` инжектит `render.yaml`. Подробности — в `DEPLOY.md`.

**Live-табло — `/dashboard/`.** Read-only для всех агентов, подтягивает изменения из репо каждые 5 секунд. Виталий и Нерсес проецируют его в зале. Опционально агент может дописать строчку в `dashboard/events.jsonl` — формат внутри `dashboard/README.md`.

**Кейсы — `/cases/`.** Бизнес-задачи в свободной форме (case_01_credit, case_02_invest). Топ говорит агенту «возьмись за это», агент сам решает к кому из соседей идти через INBOX. Никаких пошаговых playbook-ов.

## Как работает онбординг участника (из главного CLAUDE.md)

1. Скрипт `tools/cowork-onboard.py` ставит SSH-ключ из `.git/raif-workshop-key`, прописывает git config, печатает `WORKSHOP_BLOCK=<блок>` и `WORKSHOP_PARTICIPANT=<имя>`.
2. Если скрипт вернул блок — агент сразу знает кто перед ним, копирует permissions-шаблон, создаёт `<блок>/SESSION.md`, читает `<блок>/CLAUDE.md` и `INBOX/to_<блок>.md`.
3. **Шаг 7 — открывающий разговор о зоне ответственности (главный смысловой шаг воркшопа).** Агент здоровается по имени и просит топа рассказать своими словами: за что он отвечает в банке, чем занимается его блок, что у него на повестке. Рассказ агент складывает в `<блок>/state.md` как живой документ. Не навязывает обзор того, что уже есть в коде блока — топ сам должен попросить, если захочет посмотреть (или, например, поинтересоваться, что у соседей). Если попросил — агент проводит по UI блока живым языком и допишет увиденное в раздел «Что я увидел в коде блока» в том же `state.md`. Если топ хочет сразу к делу без рассказа — переходим к делу, не настаиваем.
4. Дальше — маленькие шаги, частые коммиты, регулярная проверка INBOX, бизнес-аналогии вместо жаргона. **В чате с пользователем — без markdown** (правило в `RULES.md`): только обычный человеческий текст, без хедеров, буллетов, code-fence-блоков. MD-файлы существуют в файловой системе, но сырыми не показываются.

Если `cowork-onboard.py` упал кодом 2 (старый bootstrap без ключа) — агент работает по старой схеме «спроси имя → найди в TEAM.md».

## Что считается «текущим стейтом банка» и где он живёт

Это специально устроено так, чтобы топы сами расспросили — не из markdown-описаний.

- **Стартовое состояние банка не описано ни в одном MD-файле.** В `cases/case_01_credit/README.md`, `case_02_invest/README.md`, `cases/README.md` блока «Что есть сейчас» больше нет — топы должны выяснить состояние через свой UI блока и через INBOX у соседей.
- **Состояние живёт в коде и данных.** На каждом блоке поднят минимальный сырой UI (CEO 8000, CIB 8010, Retail 8020, IT 8030, Finance 8040, Risk 8050) с реальными данными из `cases/_seed/` и со своим набором ручек. Платежи и переводы есть в Retail (между клиентами), CIB (корпоративные платежи) и Finance (внутренние переводы между балансами).
- **`<блок>/state.md` — личный документ топа.** Создаётся агентом на Шаге 7, дополняется по ходу дня. В нём только то, что сам топ сказал и что он попросил агента посмотреть. Никакого общего `bank_state.md` в `contracts/` нет и не должно появляться без явной договорённости в зале.

## Жёсткие принципы для агентов топов

- **Только своя папка.** Permissions блокируют edit чужих.
- **INBOX — единственный канал.** Не лезть в чужой код.
- **Никакого жаргона.** Никаких git/push/rebase/API/endpoint при разговоре с пользователем — только бизнес-формулировки. См. правило в `RULES.md` и пример замен в памяти.
- **Не предписывать задачи.** Топ сам решает что строить.
- **Маленькие победы.** Каждые 5–10 минут — что-то видимое.

## Тайминг подготовки (по состоянию на ~30 апреля 2026)

- 6–7 мая — оформление 1–2 страниц agenda/описания.
- 16–20 мая — тестирование среды + правка по результатам тестов.
- (Даты после самого воркшопа — пост-материалы и follow-up.)

> Сегодняшняя дата (см. env) может быть ≠ дата воркшопа. Если работаем после 3 мая — это пост-воркшоп режим: разбор полётов, follow-up, доработка для следующего запуска. Уточни у Виталия в каком режиме работаем.

## Что НЕ делать без явной просьбы

- Не редактировать `dashboard/` — он read-only для всех (включая Виталия в режиме «правлю код блока»; в режиме организатора — можно, но осторожно, на проекторе).
- Не лезть в шаблоны `.claude/templates/` без обсуждения — это влияет на то, что агенты топов смогут/не смогут делать.
- Не предлагать топам конкретные сценарии/продукты — они должны решить сами.
- Не использовать технический жаргон в текстах, которые увидят топы (README.md, INBOX-сообщения от их имени, любые слайды).
