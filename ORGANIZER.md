## Формат воркшопа

AI-воркшоп правления Райффайзен банка. Шесть членов правления делятся на
**две команды по три человека**. Обе команды получают **идентичный исходный
банк** и **параллельно, независимо** решают **одну и ту же задачу**: сначала
«добавить вкладку Кредиты», затем «добавить вкладку Инвестиции».

Внутри команды люди договариваются офлайн. Между командами связи нет —
в этом смысл соревнования: два решения одной задачи.

В банке симулируются клиенты. После каждого деплоя симулятор оценивает живой
банк команды и двигает её клиентскую базу вверх или вниз — с человеческим
обоснованием. Табло показывает счёт двух команд лицом к лицу.

## Структура репозитория

| Папка / файл | Назначение |
|---|---|
| `team_a/`, `team_b/` | Банки команд — идентичные на старте копии (FastAPI, UI, in-memory seed) |
| `simulator/` | Симулятор клиентов + табло (FastAPI, Postgres) |
| `seed/` | 500 клиентов, транзакции, кредитная история |
| `tasks/` | Брифы задач для команд (нетехнический язык) |
| `.claude/templates/settings-team-*.json` | Permissions: команда видит и правит только свою папку |
| `tools/cowork-onboard.py` | Sandbox-onboarding агента: SSH-ключ, git, `WORKSHOP_TEAM` |
| `docs/superpowers/specs/` , `docs/superpowers/plans/` | Спека дизайна и план реализации редизайна |

## Как работает симулятор клиентов

1. **Триггер — pull-модель.** Симулятор раз в ~30 секунд опрашивает `/health`
   обоих банков и читает git-коммит. Новый коммит → раунд оценки.
2. **Probe.** Закрытый список из шести HTTP-проверок банка (вкладка кредитов,
   приём заявки, синхронность решения, разделение сильный/слабый заявитель,
   человеческое объяснение отказа, регрессия переводов).
3. **Судья — рубрика + формула.** Снапшоты обеих команд уходят в LLM **одним
   вызовом** (относительная оценка честнее), `temperature=0`. LLM ставит баллы
   по 6 критериям; число клиентов считает детерминированная формула в коде
   (`B0=500`, `GAIN=0.6`). При недоступности LLM — скриптовый fallback по тем
   же checks. Симулятор не встаёт никогда.
4. **Табло** встроено в симулятор: две клиентские базы и лента событий
   с обоснованиями.

Ручное управление: `POST /admin/evaluate` (раунд по кнопке) и
`POST /admin/reset` (сброс к baseline) — с заголовком `X-Admin-Token`.

## Ручные шаги организатора

1. **Render.** Удалить 6 старых сервисов. Применить новый Blueprint
   (`render.yaml`): 3 web-сервиса + Postgres `raif-workshop-db`. В env-группе
   `ai-workshop-shared` задать `OPENAI_API_KEY` и `ADMIN_TOKEN`. Завести в
   GitHub три секрета деплой-хуков: `RENDER_HOOK_TEAM_A`, `RENDER_HOOK_TEAM_B`,
   `RENDER_HOOK_SIMULATOR`.
2. **Команды.** Распределить 6 членов правления по `team_a` / `team_b` —
   вписать в `TEAM.md` и в `$Members` в `tools/bootstrap/raif-workshop-setup.cmd`.
3. **Ноутбуки.** Прогнать обновлённый bootstrap на машинах участников, чтобы
   `.git/raif-workshop-info` содержал `WORKSHOP_TEAM=team_a|team_b`.

## Что НЕ делать

- Не давать командам подсматривать друг за другом — слепота зашита в
  `settings-team-*.json` (`Read` чужой папки запрещён).
- Не подсказывать командам реализацию — задача выдаётся брифом, решают сами
  с агентом.
