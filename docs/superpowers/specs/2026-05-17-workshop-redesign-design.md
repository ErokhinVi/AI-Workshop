# Редизайн AI-воркшопа: «два банка-команды + симулятор клиентов»

- **Дата:** 2026-05-17
- **Автор:** Claude (по заданию Виталия Ерохина)
- **Статус:** дизайн одобрен в чате, ожидает финального вычитывания спеки
- **Дедлайн реализации:** эта неделя (тестовое окно 16–20 мая 2026)

## 1. Контекст и проблема

Текущий репозиторий — монорепо AI-воркшопа для правления Райффайзен банка.
Старый формат: 6 изолированных блоков (`ceo/ cib/ retail/ it/ finance/ risk/`),
каждый — отдельный сервис на Render, коммуникация через `INBOX/`, общая
расплывчатая цель «правление строит банк», участники сами решают что делать.

Формат переделан. Новый сценарий:

- 6 членов правления делятся на **две команды по 3 человека**.
- Обе команды получают **идентичный исходный банк**.
- Обе команды **параллельно и независимо** решают **одну и ту же задачу**:
  сначала «добавить вкладку Кредиты», затем «добавить вкладку Инвестиции».
- Внутри команды люди общаются офлайн. Между командами связи нет —
  `INBOX` и `contracts` больше не нужны.
- В банке симулируются клиенты. Каждое изменение фич, выкаченное командой,
  влияет на её клиентскую базу: клиенты приходят или уходят, и это **обосновано**.
- Табло показывает счёт двух команд лицом к лицу.

Смысл — соревнование двух решений одной задачи с живой обратной связью
«фича → реакция клиентов».

## 2. Цели и не-цели

### Цели

1. Два идентичных на старте банка, разворачиваемых независимо.
2. Полная слепота команд друг к другу (агенты не читают чужую папку).
3. Симулятор: после каждого деплоя оценивает живой банк и двигает
   клиентскую базу команды с человеческим обоснованием.
4. Табло «голова к голове» с клиентскими базами и лентой событий.
5. Переписанный онбординг агента команды (нетехнический участник, задача выдаётся).
6. Брифы двух задач на нетехническом языке.
7. Решение надёжно к дедлайну «эта неделя»: скриптовый fallback симулятора,
   урезанная полировка.

### Не-цели (YAGNI для дедлайна)

- Персистентность данных банков команд (банки — in-memory на seed).
- Отдельный LLM-шлюз-сервис.
- Графики/чарты на табло сверх минимума.
- Probe и рубрика задачи 2 в первой версии — фаст-фоллоу после задачи 1.
- Аутентификация, реальные деньги, продакшен-надёжность банков.

## 4. Банк команды (`team_a/`, `team_b/`)

Каждый банк — копия `retail`-приложения: FastAPI, статический мобильный UI,
in-memory данные из `seed/`.

### Изменения относительно текущего `retail`

- **Режим только in-memory.** `DATABASE_URL` банкам не задаётся; данные грузятся
  из `seed/*.jsonl` при старте. Asyncpg-путь в коде можно оставить
  как мёртвый fallback или удалить — решаем на реализации, по умолчанию оставляем
  (меньше правок). Минус режима: тестовые мутации сбрасываются, если инстанс
  Render уснёт после 15 минут простоя. Во время активного воркшопа инстанс не
  спит; 500 клиентов перезагружаются из seed. Счёт клиентов хранит симулятор —
  он не теряется.
- **`GET /health` отдаёт git-коммит.** Поле `commit` =
  `os.environ.get("RENDER_GIT_COMMIT", "local")`. Симулятор по нему ловит деплой.
- **Хелпер `llm.py`.** Готовая функция `ask_llm(prompt: str) -> str` поверх
  OpenAI (ключ из env `OPENAI_API_KEY`). Команда вызывает функцию, не строит
  интеграцию с нуля — задача 1 требует «человеческое объяснение отказа».
- На старте — ровно одна рабочая вкладка «Переводы», `/api/credit-apply` → 501.
  Это baseline-состояние, от которого симулятор отсчитывает.

Имена папок: `team_a` / `team_b`. (Открытый пункт — см. §12.)

## 5. Симулятор клиентов (`simulator/`) — ядро редизайна

Один FastAPI-сервис. Делает четыре вещи: триггерится на деплой, снимает
состояние банка (probe), оценивает (судья), хранит счёт и отдаёт табло.

### 5.1 Триггер — pull-модель

Фоновый asyncio-цикл раз в **30 секунд** опрашивает `GET {bank}/health` обоих
банков и читает поле `commit`. Если у команды коммит отличается от последнего
оценённого (`sim_state.last_commit`) и непустой — запускается **раунд оценки**.

Раунд оценки всегда захватывает **обе** команды (см. §5.3 — честность судьи):
заново снимаются снапшоты обоих банков, делается один вызов судьи. Команда без
нового коммита получит снапшот, идентичный прошлому → дельта 0 автоматически.

Почему pull, а не push из GitHub Action: free-инстансы Render просыпаются с
холодного старта 20–30 с. Action, дёргающий симулятор после деплоя, ловил бы
старую версию или падал по таймауту. Pull-цикл сам ждёт, пока банк поднимется
с новым коммитом; debounce встроен — оценка только на смену коммита.

Цикл устойчив к ошибкам: недоступный `/health` (timeout, 502/503) — не падение,
а пропуск тика; для оценки «банк недоступен» см. §5.4.

### 5.2 Probe — снятие состояния банка (закрытый список)

Probe задачи 1 — фиксированная последовательность HTTP-вызовов к живому банку.
Фиксированные ID берутся из `seed/`: один **сильный заявитель** (premium,
высокий доход, без просрочек) и один **слабый** (mass, низкий доход, история
просрочек) — конкретные ID выбираются при реализации и хардкодятся в probe.

| # | Вызов | Что извлекается |
|---|---|---|
| P1 | `GET /health` | 200; `commit`, число клиентов; доступность банка |
| P2 | `GET /` | 200; HTML. Содержит ли (регистронезависимо) «кредит»; содержит ли «перевод» (регрессия UI) |
| P3 | `POST /api/credit-apply` `{client_id: <сильный>, amount_rub: 300000, term_months: 12}` | HTTP-статус (200 / 501 / 500 / 4xx); латентность мс; есть ли в ответе поле решения; вердикт |
| P4 | `POST /api/credit-apply` `{client_id: <слабый>, amount_rub: 900000, term_months: 6}` | Статус; вердикт; есть ли непустое человеческое объяснение (> 40 символов, не «rejected») |
| P5 | `GET /credit-applications` | 200; список; видны ли две только что поданные заявки |
| P6 | `POST /api/transfer` `{from: <клиент>, to: <клиент>, amount_rub: 1000}` | 200 — переводы не сломаны (регрессия) |

Снапшот — структурированный JSON:

```json
{
  "team": "team_a",
  "commit": "abc123",
  "ts": "2026-05-17T12:00:00Z",
  "reachable": true,
  "checks": {
    "credit_mentioned_in_ui": true,
    "credit_apply_status": 200,
    "credit_apply_latency_ms": 1200,
    "credit_response_has_decision": true,
    "credit_response_has_explanation": true,
    "decision_strong": "approved",
    "decision_weak": "rejected",
    "decision_is_discriminating": true,
    "credit_applications_listed": true,
    "transfer_regression_ok": true,
    "transfer_mentioned_in_ui": true
  },
  "raw": { "...": "усечённые тела ответов для судьи" }
}
```

`decision_is_discriminating` = сильный и слабый получили разные вердикты —
прокси «решение опирается на данные клиента, а не рандом».

Probe задачи 2 («Инвестиции») — отдельный набор, фаст-фоллоу. Probe устроен
как подключаемый модуль на активную задачу (`ACTIVE_TASK` env: `credit` / `invest`).

### 5.3 Судья — рубрика плюс формула

**Рубрика задачи 1** — 6 критериев, каждый 0–2 балла (макс. 12):

| Критерий | Описание |
|---|---|
| C1 | Вкладка «Кредиты» присутствует в интерфейсе |
| C2 | Заявка отправляется, сервер отвечает без ошибки (200, не 500/501) |
| C3 | Решение синхронное — ответ за < 5 секунд |
| C4 | Решение опирается на данные клиента — сильный и слабый заявители получают разные вердикты |
| C5 | Отказ сопровождается осмысленным человеческим объяснением |
| C6 | Нет регрессии — вкладка «Переводы» и сам перевод работают |

**Вызов LLM-судьи.** Снапшоты обеих команд уходят в LLM **одним вызовом** —
модель видит их рядом, относительная оценка честнее. `temperature=0`. LLM
возвращает строгий JSON:

```json
{
  "team_a": { "scores": [2,2,2,0,0,2], "reason": "короткое человеческое обоснование" },
  "team_b": { "scores": [2,2,1,2,2,2], "reason": "..." }
}
```

LLM **не** выдаёт число клиентов. Он ставит баллы по фиксированным критериям и
пишет короткое (1–2 предложения) обоснование, привязанное к фактам снапшота.
Число клиентов считает детерминированная формула в коде.

**Формула баллы → клиенты.** Параметры (значения по умолчанию, тюнятся):

- `B0 = 500` — стартовая база каждой команды.
- `S` — сумма баллов рубрики, `[0, 12]`.
- `S_base` — балл baseline-снапшота нетронутого банка; замеряется симулятором
  на старте и сохраняется (ожидаемо ≈ 2: работает только «Переводы»).
- `S_prev` — балл прошлой оценки команды.
- `GAIN = 0.6`.

**Инициализация.** При первом запуске симулятор делает baseline-раунд: probe
обоих (ещё нетронутых) банков, сохраняет `S_base`, выставляет `S_prev := S_base`,
`client_base := B0`, `last_commit :=` текущий коммит каждого банка. То же делает
`POST /admin/reset`.

Алгоритм раунда для команды:

1. Probe → снапшот, `S` от судьи (или fallback, §5.4).
2. Если `S == S_prev` → `delta = 0`, причина «изменения не повлияли на оценку
   банка». (Чистый no-op деплой даёт ноль автоматически.)
3. Иначе:
   `I = clamp((S − S_base) / 12, −1, 1)`
   `target = round(B0 × (1 + GAIN × I))`
   `delta = target − client_base`
   `client_base = target`
4. `client_base` зажимается снизу значением 50 (банк всегда с клиентами).
5. `S_prev := S`, событие пишется в журнал.

Worked example (`B0=500, S_base=2, GAIN=0.6`):
нетронутый банк `S=2` → `I=0` → target 500.
вкладка есть, заявка работает синхронно, но без объяснения и без разделения
сильный/слабый (`C1,C2,C3,C6=2`, `C4,C5=0`) → `S=8` → `I=0.5` → target 650 (+150 от 500).
полностью готовая задача → `S=12` → `I=0.83` → target 750 (ещё +100).
сломали «Переводы» (`C6→0`) → `S` падает ниже `S_base` → target ниже 500 → отрицательная дельта.

### 5.4 Скриптовый fallback

Воркшоп на этой неделе — права на срыв нет. Если LLM недоступен или вернул не-JSON,
баллы рубрики заполняются механически из `checks` снапшота:

- C1 = `credit_mentioned_in_ui` ? 2 : 0
- C2 = `credit_apply_status==200` ? 2 : 0
- C3 = латентность < 5000 → 2; < 10000 → 1; иначе 0
- C4 = `decision_is_discriminating` ? 2 : 0
- C5 = `credit_response_has_explanation` ? 2 : 0
- C6 = `transfer_regression_ok` ? 2 : 0

Обоснование собирается из шаблонов по баллам. Формула §5.3 — та же. Симулятор
не встаёт никогда. Поле `judge` в событии помечает `llm` или `fallback`.

**Банк недоступен** (`reachable=false`): особый случай — `target = round(client_base × 0.8)`,
причина «банк не открывается, клиенты не могут войти».

### 5.5 Хранение (Postgres, одна схема)

Postgres нужен **только симулятору** — счёт и журнал. Один free Postgres-инстанс
Render, одна схема. (Free Postgres живёт 90 дней — создать свежим перед событием.)

```sql
sim_state (
  team          TEXT PRIMARY KEY,      -- 'team_a' | 'team_b'
  client_base   INT  NOT NULL,
  last_commit   TEXT,                  -- последний оценённый коммит
  baseline_score INT,                  -- S_base
  last_score    INT,                   -- S_prev
  updated_at    TIMESTAMPTZ
)
sim_events (
  id            BIGSERIAL PRIMARY KEY,
  team          TEXT,
  ts            TIMESTAMPTZ,
  commit        TEXT,
  delta         INT,
  client_base_after INT,
  rubric        JSONB,                 -- 6 баллов
  reason        TEXT,
  snapshot      JSONB,                 -- checks
  judge         TEXT                   -- 'llm' | 'fallback'
)
```

### 5.6 Эндпоинты симулятора

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/` | HTML-табло |
| GET | `/state` | JSON: базы обеих команд + последние события |
| GET | `/health` | здоровье самого симулятора |
| POST | `/admin/evaluate` | ручной раунд оценки (кнопка организатора), токен |
| POST | `/admin/reset` | сброс баз к `B0`, очистка журнала, ре-baseline, токен |

`/admin/*` защищены токеном из env `ADMIN_TOKEN`. Ручной `/admin/evaluate`
нужен для локального теста (где `commit=local` не меняется) и для «раунда по
кнопке» в зале.

### 5.7 Табло

`GET /` симулятора — HTML-страница, авто-обновление через `fetch /state` раз в
несколько секунд. Минимум без чарт-библиотек:

- Две колонки: Команда A / Команда B.
- Большое число — текущая клиентская база, стрелка последней дельты.
- Простой инлайн-SVG-спарклайн тренда (по желанию, если успеваем).
- Лента событий: время, команда, дельта, обоснование, метка судьи (llm/fallback).

Старый `dashboard/` (скан git-лога и INBOX) удаляется целиком — табло теперь
читает состояние из симулятора.

## 6. База данных — ответ на «пересоздавать ли»

Да, пересоздаём, радикально проще текущего. Postgres — только у симулятора
(§5.5). Банки команд — in-memory на seed, к БД не прикасаются. Это убирает
изоляцию двух команд в одной базе через схемы и снимает с команд-нетехнарей
любой контакт с БД. Один free Postgres-инстанс, одна схема. Если позже
понадобится персистентность банков — это отдельная история со схемами и
`search_path`, в текущий объём не входит.

## 7. Render и деплой

Было 6 web-сервисов → станет **3**: `raif-team-a`, `raif-team-b`, `raif-simulator`.

`render.yaml` переписывается:

- 3 сервиса, каждый из своего Dockerfile, `buildFilter` по своей папке.
- Env-группа `ai-workshop-shared`: `OPENAI_API_KEY`, `OPENAI_BASE_URL`,
  `OPENAI_MODEL` — всем трём (банкам для объяснения отказа, симулятору для судьи).
  `DATABASE_URL` — только симулятору. `ADMIN_TOKEN` — симулятору.
- Симулятору — `BANK_A_URL`, `BANK_B_URL` (публичные URL банков), `ACTIVE_TASK`.
- Банкам — `RENDER_GIT_COMMIT` отдаёт Render автоматически.

`.github/workflows/deploy-render.yml` упрощается: diff папок → деплой-хук только
тронутого сервиса (`team_a/**` → `raif-team-a` и т.д.). Симулятор в Action не
дёргается — он на pull-модели. Секреты деплой-хуков: `RENDER_HOOK_TEAM_A`,
`RENDER_HOOK_TEAM_B`, `RENDER_HOOK_SIMULATOR`.

Отдельный LLM-шлюз не делаем — в банк встроен хелпер `llm.py` (§4).

## 8. Слепота команд

`.claude/templates/settings-team-a.json`:

- `Edit`/`Write` разрешены только в `team_a/**`.
- `Read` запрещён на `team_b/**` — агент команды A физически не видит код B.
- `Edit` запрещён на `team_b/**`, `simulator/**`, `seed/**`, `render.yaml`,
  `.github/**`, корневые служебные файлы.
- `Read` разрешён на `tasks/**`, `seed/**`, свою папку. `simulator/**` — закрыт
  (probe и рубрика — инструмент организатора; требования к команде несёт
  нетехнический бриф задачи).

`settings-team-b.json` — зеркально. Без слепоты к середине дня агенты подсмотрят
решения друг друга и оба банка станут одинаковыми — соревнование развалится.

## 9. Онбординг (переписанный `CLAUDE.md`)

Радикально проще старого. Агент команды:

1. **Шаг 0** — `tools/cowork-onboard.py` (починенный): SSH-ключ, git config,
   читает `WORKSHOP_TEAM` из `.git/raif-workshop-info`, печатает
   `WORKSHOP_TEAM=team_a|team_b`.
2. Поздороваться, узнать участника и команду (по `TEAM.md`).
3. Скопировать `settings-team-X.json` → `.claude/settings.local.json` (слепота).
4. Прочитать бриф активной задачи `tasks/task_01_credit.md`.

## 14. Риски

- **Холодный старт Render free** (20–30 с) — pull-цикл симулятора терпит
  таймауты и 502/503, не падает.
- **Отказ LLM-судьи** — скриптовый fallback (§5.4).
- **Free Postgres живёт 90 дней** — создать свежим перед событием.
- **Команда ломает свой банк** — `reachable=false` обрабатывается особым
  случаем (§5.4), симулятор не падает.
- **Две команды, одна ветка `main`** — папки команд не пересекаются, конфликтов
  нет; правило `pull --rebase` перед push сохраняется.
- **Дедлайн «эта неделя»** — фазовый план: Фазы 0–7 = минимально рабочий
  воркшоп, Фаза 8 = фаст-фоллоу.
